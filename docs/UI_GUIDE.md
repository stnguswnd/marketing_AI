# UI 디자인 가이드

## 구현 전제
- UI는 `frontend/` 아래 Next.js App Router로 구현한다.
- 점주용/관리자용 라우트 그룹을 나누고, 공통 레이아웃과 디자인 토큰은 `frontend/` 내부에서 공유한다.
- Python 백엔드의 `backend/app/` 디렉토리는 API 전용이므로 프론트 라우트 루트로 사용하지 않는다.

## 디자인 원칙
1. 챗봇이 아니라 **실무용 운영 툴**처럼 보여야 한다.
2. 점주 화면은 **입력 피로를 줄이는 것**, 관리자 화면은 **판단 속도를 높이는 것**이 핵심이다.
3. 모든 주요 액션은 **현재 상태 / 다음 단계 / 책임 주체**가 분명해야 한다.
4. 생성 결과를 과장하지 말고, **검토 가능성 / 승인 가능성 / 이력 추적성**을 먼저 보여준다.
5. 장식보다 **정보 위계, 정렬, 여백, 타이포 가독성**을 우선한다.

## 제품 톤
- 점주 화면: 짧은 단계, 큰 입력 영역, 쉬운 용어, 강한 1차 CTA
- 관리자 화면: 테이블 중심, 상태 배지 중심, 빠른 필터링과 스캔
- 전체 인상: 무신사/29CM 계열의 **단정한 커머스형 SaaS**
- 말투: 친절하지만 건조하게, 과장 없이 명확하게
- 시각 톤: 흑백 기반, 절제된 포인트 컬러, 강한 타이포 대비

## UX 원칙
- 콘텐츠 생성은 채팅 입력창이 아니라 **모달 또는 스텝 폼**으로 시작한다.
- 필수 입력은 최소화하되, 품질과 승인에 필요한 값은 반드시 구조화해서 받는다.
- `"생성"`과 `"게시"`는 절대 같은 버튼으로 합치지 않는다.
- 상태값은 색상만 쓰지 말고 **텍스트 + 배지 + 위치**로 함께 보여준다.
- 리뷰 대응, 게시 요청, 리포트 생성은 각각 **독립된 작업 흐름**으로 구분한다.
- 사용자가 지금 무엇을 해야 하는지보다, **무엇을 기다리는 상태인지**도 명확히 보여준다.

## AI 슬롭 안티패턴
| 금지 사항 | 이유 |
|-----------|------|
| 전체 화면 챗 인터페이스 | 구조화 입력 중심 제품 전략과 맞지 않음 |
| "Ask AI anything" 같은 범용 카피 | 사용 범위가 흐려지고 업무 도구 정체성이 약해짐 |
| 과한 글래스모피즘 / blur / neon | 실무형 툴의 신뢰감을 떨어뜨림 |
| 보라 / 인디고 중심 AI 브랜딩 | 제품 문제 해결과 무관한 클리셰 |
| 의미 없는 생성 애니메이션 | 처리 상태보다 장식이 앞서 보임 |
| 모든 카드와 버튼이 같은 시각적 무게 | 정보 위계가 무너져 스캔 속도가 떨어짐 |
| 설명 없는 아이콘-only 액션 | 운영 화면에서 오작동 위험이 큼 |

## 색상

### 기본 컬러 전략
- 전체 색감은 **블랙 / 화이트 / 그레이** 중심으로 가져간다.
- 포인트 컬러는 최소 1~2개만 사용한다.
- 브랜드 포인트는 화려함보다 **명확한 대비** 역할만 한다.
- 관리자 화면에서는 색보다 **배지 텍스트와 테이블 정렬**이 먼저 읽혀야 한다.

## 타이포그래피
- 기본 폰트는 `Pretendard`, 대체 폰트는 시스템 산세리프를 사용한다.
- 타이포의 역할은 “분위기”보다 **구조와 우선순위 표시**에 있다.
- 본문은 짧고 밀도 있게, 제목은 과장 없이 단단하게 간다.

| 용도 | 스타일 |
|------|--------|
| 페이지 제목 | `text-[30px] font-semibold tracking-[-0.02em] text-[#111418]` |
| 섹션 제목 | `text-[18px] font-semibold text-[#111418]` |
| 카드 제목 | `text-[14px] font-semibold text-[#1B2430]` |
| 본문 | `text-[14px] leading-6 text-[#4B5563]` |
| 라벨 | `text-[11px] font-semibold uppercase tracking-[0.08em] text-[#6B7684]` |
| 테이블 셀 | `text-[13px] text-[#2F3944]` |
| 상태 배지 | `text-[11px] font-semibold tracking-[0.02em]` |

## 레이아웃
- 전체 레이아웃은 **좌측 정렬**을 기본으로 한다.
- 점주 화면은 `max-w-4xl` 기준의 **단일 작업 흐름**에 맞춘다.
- 관리자 화면은 `max-w-7xl` 이상에서 **필터 + 테이블 + 상세 패널** 구조를 우선한다.
- 페이지는 넓게 쓰되, 카드 내부는 촘촘하게 정리한다.
- 섹션 간 간격은 분명하게 두고, 같은 블록 안에서는 간격을 일정하게 유지한다.
- 모바일에서는 테이블을 억지로 유지하지 말고 **상태 요약 카드**로 재구성한다.

## 컴포넌트 가이드

### 페이지 구조
- 상단에는 항상 **현재 작업 맥락**, **현재 상태**, **다음 액션**을 먼저 보여준다.
- 한 화면에서 강한 CTA는 하나만 둔다.
- 보조 액션은 보조 버튼 또는 텍스트 버튼으로 낮춘다.
- 점주용 생성 화면에서는 결과 미리보기보다 **입력 폼**이 먼저 와야 한다.
- 관리자 화면에서는 KPI 카드보다 **실제 처리해야 할 큐와 표**를 먼저 보여준다.

### 모달 / 폼
- 1개 모달에는 5~7개 수준의 핵심 입력만 담는다.
- 선택형 필드는 드롭다운, 세그먼트, 라디오로 명확히 나눈다.
- 필수 / 선택 / 권장값은 시각적으로 구분한다.
- 업로드 영역은 파일 개수, 업로드 완료 여부, 미리보기 상태를 바로 보여준다.
- 생성 전에는 “입력 요약” 블록을 두어 누락을 줄인다.
- 폼은 친절해야 하지만 길어 보이면 안 된다.

### 테이블 / 운영 화면
- 승인 대기, 게시 완료, 리뷰 민감도, 담당자, 최근 수정일 컬럼은 우선 노출한다.
- 행 액션은 즉시 실행보다 **상세 보기 / 검토 진입**을 우선한다.
- 필터는 국가, 플랫폼, 상태, 기간, 담당자 기준으로 고정 제공한다.
- 테이블은 줄무늬보다 **보더, 정렬, 여백**으로 읽히게 만든다.
- 행 hover는 과하지 않게 배경색 한 톤만 바꾼다.

### 카드
```text
rounded-2xl border border-[#E3E8EE] bg-white px-5 py-5 shadow-[0_1px_2px_rgba(16,24,40,0.04)]
```

### 버튼
```text
Primary: rounded-lg bg-[#111111] px-4 py-2.5 text-white hover:bg-[#000000]
Secondary: rounded-lg border border-[#D7DDE5] bg-white px-4 py-2.5 text-[#1F2937] hover:bg-[#F8FAFC]
Text: text-[#4B5563] hover:text-[#111418]
Danger: rounded-lg bg-[#C0392B] px-4 py-2.5 text-white hover:bg-[#A93226]
```

### 입력 필드
```text
rounded-lg border border-[#D7DDE5] bg-white px-4 py-3 text-[#111418] placeholder:text-[#9AA4B2]
focus:border-[#111111] focus:ring-2 focus:ring-black/10
```

### 상태 배지
```text
Draft: bg-[#F3F4F6] text-[#4B5563]
Approved: bg-[#E8F7EF] text-[#167C53]
Scheduled: bg-[#EAF2FF] text-[#2563EB]
Published: bg-[#E7F7F4] text-[#0F766E]
Rejected: bg-[#FDECEC] text-[#C0392B]
Escalated: bg-[#FEF3E6] text-[#A16207]
```

## 화면별 가이드

### 점주 화면
- 첫 화면에서 **지금 해야 할 일**이 바로 보여야 한다.
- 콘텐츠 생성은 `"나라 → 플랫폼 → 목적 → 입력 내용 → 자산 첨부"` 순으로 자연스럽게 이어져야 한다.
- 복잡한 마케팅 용어보다 쉬운 표현을 사용한다.
- 생성 결과는 문장 품질 자랑보다 **초안 / 승인 대기 / 반려 사유 / 다음 액션** 중심으로 설명한다.
- “AI가 대신 해줬다”보다 “이제 검토하고 승인하면 된다”가 먼저 보여야 한다.

### 관리자 화면
- 사업 전체 현황, 점포별 현황, 승인 큐, 민감 리뷰 큐를 분리해서 보여준다.
- KPI 카드는 상단 요약 정도로만 쓰고, 핵심은 **운영 테이블과 처리 큐**다.
- 감사 로그, 버전 이력, 승인 히스토리는 1~2번 클릭 안에 접근 가능해야 한다.
- 점포 단위 상세 페이지에서는 “최근 생성물 / 반려 사유 / 담당자 액션”이 먼저 보여야 한다.

## 애니메이션
- 허용: `fade-in` 0.2~0.25초
- 허용: 모달 진입/이탈
- 허용: 저장 완료 토스트
- 허용: 비동기 처리 상태를 보여주는 짧은 progress indicator
- 금지: 큰 배경 모션, 반복 글로우, 불필요한 shimmer, 장식용 로딩

## 아이콘
- 아이콘은 단색 SVG를 사용한다.
- `strokeWidth`는 `1.5` 또는 `1.75`로 통일한다.
- 상태 표현은 아이콘만으로 전달하지 않고 텍스트를 함께 둔다.
- 아이콘 배경용 원형 칩은 꼭 필요한 경우만 사용한다.
- 관리자 화면에서는 아이콘보다 **텍스트 라벨과 정렬**이 먼저 읽혀야 한다.

## 접근성
- 모든 폼 필드는 라벨과 에러 메시지를 명시적으로 연결한다.
- 상태는 색상만으로 의미를 전달하지 않는다.
- 키보드만으로 모달 열기, 필드 이동, 제출이 가능해야 한다.
- 승인 / 반려 / 게시 액션은 확인 문구를 포함해 오작동을 줄인다.
- 테이블은 헤더, 정렬 기준, 필터 상태를 명확히 드러낸다.

## 구현 메모
- 기본 폰트는 `Pretendard` 사용
- 카드 radius는 너무 둥글지 않게 `xl~2xl` 수준 유지
- 그림자는 약하게, 보더는 분명하게
- 브랜드 포인트는 블랙을 중심으로 하고, 정보성 액션에만 블루를 제한적으로 사용
- 전체 UI는 “AI 툴”보다 “무신사스러운 운영형 커머스 백오피스”에 가깝게 유지
