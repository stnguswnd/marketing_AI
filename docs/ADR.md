# Architecture Decision Records

## 철학
MVP 속도와 실제 운영 가능성을 동시에 잡는다.  
이 프로젝트는 챗봇형 데모가 아니라, 소상공인의 글로벌 마케팅 업무를 실제로 줄여주는 운영형 SaaS를 목표로 한다.

사용자 입력은 자유 자연어보다 구조화된 폼을 우선하고, AI는 그 뒤에서 실행되는 내부 오케스트레이터로 사용한다.  
외부 자동화 툴에 과도하게 의존하지 않고, Python 코드 기반으로 핵심 로직을 통제 가능하게 설계한다.

AI는 "멋진 생성"보다 "업무 대체, 승인 가능성, 감사 가능성, 운영 안정성"을 우선한다.

### ADR-001: n8n 대신 Python 중심 오케스트레이션 선택
**결정**: 핵심 워크플로우는 FastAPI + LangGraph + Celery + Redis로 구현한다.

**이유**: 이 프로젝트는 단순 트리거 자동화가 아니라 상태 저장, 분기 처리, 승인, 재시도, 장시간 실행이 중요한 구조다. Python 중심으로 구현하면 서비스 로직을 세밀하게 통제할 수 있고, 테스트와 유지보수 범위도 명확해진다.

**트레이드오프**: n8n처럼 비개발자가 시각적으로 플로우를 수정하기는 어렵다. 초기 구현 난이도는 조금 올라간다.

**완화 방안**:
- 초기 MVP에서는 모든 워크플로우를 다 만들지 않고, 점포 온보딩 / 콘텐츠 생성 / 승인 플로우 3개만 우선 구현한다.
- 운영자가 자주 바꿔야 하는 정책은 코드 하드코딩 대신 DB 설정값 또는 YAML/JSON 설정 파일로 분리한다.
- 외부 자동화 도구가 꼭 필요한 구간은 전면 배제하지 않고, 알림/CRM 동기화처럼 비핵심 영역에 한해 추후 연결 가능하도록 인터페이스를 남겨둔다.

**재검토 시점**:
- 운영자가 직접 플로우를 수정해야 하는 요청이 반복적으로 발생할 때
- 비개발 인력이 운영 자동화 수정에 참여해야 할 때
- Python 기반 워크플로우 유지보수 비용이 지나치게 커질 때

### ADR-002: 사용자 입력은 챗형 자연어가 아니라 구조화된 폼 입력을 기본으로 한다
**결정**: 점주는 자연어 요청 대신 나라 선택, 플랫폼 선택, 목적, 입력 내용, 홈페이지 URL, 이미지 업로드 등의 모달/폼 UI를 통해 입력한다.

**이유**: 입력 품질을 일정하게 유지하고 누락값을 줄여야 승인, 게시, 리포트, 대시보드까지 이어지는 운영 자동화를 안정적으로 만들 수 있다. 구조화된 입력은 설명 가능성과 통제 가능성도 높다.

**트레이드오프**: 자유도가 높은 대화형 UX는 줄어든다. 대신 서비스 신뢰성과 운영 일관성이 높아진다.

**완화 방안**:
- 구조화된 필드를 기본으로 하되, “추가 요청사항” 자유 입력란을 보조적으로 제공한다.
- 반복 입력이 많은 필드는 점포 프로필에서 자동 prefill 한다.
- 국가/플랫폼별 추천값과 예시 문구를 UI에서 제공해 폼 입력 부담을 줄인다.
- 장기적으로는 구조화된 폼 입력을 채워주는 보조형 AI 입력 도우미를 붙일 수 있도록 확장 여지를 남긴다.

**재검토 시점**:
- 사용자가 폼 입력을 지나치게 답답해하거나 이탈률이 높을 때
- 반복적으로 “직접 말로 요청하고 싶다”는 요구가 쌓일 때
- 입력 필드가 너무 많아져 폼 기반 UX의 장점이 줄어들 때

### ADR-003: LangGraph 기반 상태형 AI 워크플로우 선택
**결정**: AI 실행 흐름은 LangGraph 기반의 bounded workflow로 설계한다.

**이유**: 점포 온보딩, 콘텐츠 생성, 리뷰 대응, 리포트 생성은 모두 상태 관리와 중간 승인, 실패 후 재개가 중요하다. 단순 체인보다 노드 기반 상태형 그래프가 적합하다.

**트레이드오프**: 단순 REST API 호출 구조보다 설계가 복잡해진다. 디버깅 체계와 상태 스키마 설계가 필수다.

**완화 방안**:
- 그래프는 처음부터 범용적으로 만들지 않고, use case 단위로 분리한다. (`onboarding_graph`, `content_graph`, `review_graph`)
- 각 노드의 입력/출력을 TypedDict 또는 Pydantic 스키마로 강제한다.
- 노드별 단위 테스트와 그래프 통합 테스트를 분리한다.
- 상태 전이를 로깅하고, 실패 지점과 재시도 이력을 추적할 수 있도록 audit log와 execution log를 남긴다.

**재검토 시점**:
- 실제 워크플로우가 단순해져 stateful graph의 이점이 사라질 때
- LangGraph가 과도한 복잡도를 만들고 있다는 신호가 나올 때
- 반대로 승인/재개/장기 실행 요구가 더 커져 Temporal 같은 상위 워크플로우 엔진 검토가 필요할 때

### ADR-004: PostgreSQL 중심 통합 저장소 선택
**결정**: 정형 데이터, JSON 기반 브랜드 프로필, 감사 로그, 작업 이력은 PostgreSQL 중심으로 저장한다.

**이유**: 점포 정보, 승인 이력, 콘텐츠 버전, 리뷰 이벤트, 보고서 생성 기록 등 운영 데이터가 많다. MVP 단계에서는 저장소를 지나치게 분리하기보다 Postgres 중심으로 일원화하는 것이 유지보수와 추적에 유리하다.

**트레이드오프**: 초기에는 완전한 벡터 검색 인프라나 대규모 분석 파이프라인까지 포함하지 않는다. 트래픽과 데이터 규모가 커지면 검색/분석 계층 분리가 필요할 수 있다.

**완화 방안**:
- 초기에 스키마를 과도하게 쪼개지 않고, 확장 가능하도록 JSONB 필드를 병행 사용한다.
- 검색이 중요해지는 시점까지는 PostgreSQL FTS와 기본 인덱스로 버틴다.
- 벡터 검색이 꼭 필요해지면 pgvector를 우선 검토하고, 그 이후에만 별도 검색 엔진을 분리한다.
- 리포팅 부담이 커지면 OLTP DB와 별도로 read replica 또는 분석 전용 저장소를 분리하는 로드맵을 문서에 명시한다.

**재검토 시점**:
- 검색 응답 속도나 정확도가 제품 경험을 해치기 시작할 때
- 리포트/집계 쿼리 때문에 운영 DB 부하가 커질 때
- 벡터 검색과 분석 수요가 늘어나 단일 DB 구조가 병목이 될 때

### ADR-005: 승인 기반 발행 구조 선택
**결정**: AI가 생성한 외부 게시물과 민감한 리뷰 답변은 자동 게시하지 않고 승인 단계를 둔다.

**이유**: B2G/소상공인 환경에서는 품질 그 자체보다 통제 가능성과 책임 추적이 중요하다. 승인 구조가 있어야 환각, 오역, 부적절한 표현 리스크를 줄일 수 있다.

**트레이드오프**: 완전 자동화 속도는 떨어진다. 대신 신뢰성과 실서비스 적합성이 높아진다.

**완화 방안**:
- 모든 항목을 동일하게 승인하지 않고, 위험도 기반 승인 정책을 둔다.
- 저위험 작업(예: 내부 초안, 단순 긍정 리뷰 답글)은 자동 draft 생성까지만 허용한다.
- 고위험 작업(외부 게시, 부정 리뷰, 기관 제출 리포트)만 명시적 승인 대상으로 둔다.
- 승인 UI에서 수정 후 승인 기능을 제공해 운영자의 재작업 비용을 줄인다.

**재검토 시점**:
- 승인 대기 시간이 실제 운영 병목이 될 때
- 반복적으로 동일한 유형의 콘텐츠가 문제 없이 통과될 때
- 충분한 운영 데이터가 쌓여 일부 자동 승인 정책을 안전하게 도입할 수 있을 때

### ADR-006: 멀티에이전트보다 bounded node system 선택
**결정**: 초기 MVP는 완전자율 멀티에이전트 대신 역할이 고정된 노드 기반 시스템으로 구현한다.

**이유**: 초기 단계에서 멀티에이전트는 품질 편차와 디버깅 난이도를 높인다. 노드별 입력/출력을 고정하면 코드 생성, 테스트, 운영 추적이 쉬워진다.

**트레이드오프**: 에이전트 자율성은 줄어든다. 대신 예측 가능성과 설명 가능성이 높아진다.

**완화 방안**:
- 노드 경계를 너무 잘게 쪼개지 않고, 실제 업무 단위로 묶는다. (전략 수립 / 카피 생성 / 검수)
- 반복적으로 패턴이 확인되면 그때만 일부 노드를 서브에이전트화한다.
- 초기에는 bounded workflow로 운영하고, 향후 성숙도에 따라 제한적 agent delegation을 도입하는 로드맵을 남긴다.

**재검토 시점**:
- 업무 유형이 늘어나 고정 노드 구조만으로 대응이 어려워질 때
- 특정 노드가 지나치게 커져 사실상 내부에 여러 역할을 품게 될 때
- 자율적 플래닝과 툴 선택이 실제 성능 향상으로 이어진다는 근거가 확보될 때

### ADR-007: 외부 플랫폼 연동은 어댑터 계층으로 강제한다
**결정**: 소셜, 리뷰, 메시징, LLM 연동은 모두 `integrations/` 하위 어댑터를 통해서만 접근한다.

**이유**: 서비스 레이어와 외부 SDK를 직접 결합하면 테스트가 어려워지고 공급자 교체 비용이 커진다. 어댑터 계층을 두면 mock 테스트, 재시도 정책, 인증 관리, 장애 격리가 쉬워진다.

**트레이드오프**: 추상화 코드가 추가된다. 초기에는 구현량이 늘어나지만 장기적으로 변경 비용이 줄어든다.

**완화 방안**:
- 모든 공급자에 대해 과도한 공통 인터페이스를 만들지 않고, 실제 필요한 기능만 최소 단위로 추상화한다.
- 초기 MVP에서는 우선순위가 높은 플랫폼만 어댑터 구현 대상으로 제한한다. (예: Instagram, Google Business, Email)
- 어댑터별 mock/fake 구현을 함께 두어 테스트 효율을 높인다.
- 인증, 재시도, rate limit 대응은 공통 유틸로 묶고, 플랫폼별 차이는 어댑터 내부에서 처리한다.

**재검토 시점**:
- 특정 공급자에 강하게 종속되어 추상화 이점이 거의 없어질 때
- 어댑터 계층이 과도한 중복과 복잡도를 만들기 시작할 때
- 새로운 플랫폼 추가 속도가 너무 느려져 아키텍처가 오히려 제품 확장을 막을 때

### ADR-008: API 라우터는 얇게 유지하고 비즈니스 로직은 서비스 계층에 집중한다
**결정**: FastAPI 라우터는 요청/응답 변환과 권한 확인만 담당하고, 실제 업무 로직은 `services/`와 `graphs/`에 둔다.

**이유**: 라우터에서 비즈니스 로직이 커지면 테스트 경계가 흐려지고 재사용성이 떨어진다. 서비스 계층을 중심으로 두면 API, 배치, 워커가 같은 로직을 공유할 수 있다.

**트레이드오프**: 파일 수와 레이어 수가 증가한다. 그러나 운영형 SaaS에서 유지보수성과 테스트성 이점이 더 크다.

**완화 방안**:
- 초기에는 과도한 레이어 분리를 피하고, `router → service → repository` 3단 구조를 기본으로 유지한다.
- 서비스 계층이 지나치게 비대해지면 도메인 단위로 서비스 파일을 분리한다.
- 공통 비즈니스 로직만 서비스에 두고, 단순 포맷 변환은 라우터 또는 schema 계층에서 처리한다.
- 신규 기능 추가 시 “이 로직이 API 외에도 재사용될 가능성이 있는가?”를 기준으로 서비스 이동 여부를 판단한다.

**재검토 시점**:
- 서비스 계층이 지나치게 비대해져 또 다른 god-object가 되기 시작할 때
- 반대로 대부분의 로직이 API 요청 한 번에만 쓰여 레이어 비용이 과하다고 느껴질 때
- 워커/배치/API 간 로직 공유가 예상보다 적어 구조 단순화 필요성이 커질 때

## ADR 운영 원칙
- ADR은 한 번 쓰고 끝나는 문서가 아니라, 제품 단계 변화에 따라 갱신한다.
- 각 ADR의 재검토 시점이 발생하면 유지 / 수정 / 폐기 여부를 기록한다.
- 초기 MVP 기준의 결정과 파일럿 이후의 결정은 구분해서 관리한다.
- 새로운 기술 도입은 기존 ADR과 충돌하는지 먼저 검토한다.
